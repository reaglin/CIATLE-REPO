## 1. Overview

The PreseMaker Community Repository is a publicly browsable, contributor-maintained library of course materials generated by the PreseMaker authoring tool. Materials are organized using a configurable, hierarchical taxonomy defined at repository setup. The repository is designed to be adopted by any organization — schools, school districts, companies, or consortia — that needs a structured, searchable library of instructional materials.

The **Florida Statewide Course Numbering System (SCNS)** serves as the reference taxonomy in this document and in the reference implementation. All taxonomy examples use SCNS unless otherwise noted. The system is taxonomy-agnostic and supports any organization's course structure provided it conforms to the three-level hierarchy described in Section 3.

The repository serves three primary functions:

- **Publish** — contributors upload materials at the course, module, or material level, associated with a valid course identifier in the configured taxonomy.
- **Browse** — users discover and download materials published by other contributors, usable as starting points or supplementary resources.
- **Organize** — the repository maintains a consistent structure so that materials for the same course, regardless of contributor or institution, are always co-located and easily discoverable.

## 3. Taxonomy Model

### 3.1 Structure

The repository taxonomy is a three-level hierarchy. Every node at each level has a short key used in identifiers and URLs, and a display name. The maximum depth is three levels.

```
Level 1  (e.g., Discipline)
  └── Level 2  (e.g., Subdiscipline)
        └── Level 3  (e.g., Subject Area / Prefix)
              └── Courses
```

**SCNS Reference Example:**
```
Engineering Technologies               ← Level 1
  └── Electronic Engineering           ← Level 2
        └── EET                        ← Level 3 (Prefix)
              └── EET1084C             ← Course
```

**Custom Taxonomy Example (corporate training):**
```
Operations                             ← Level 1
  └── Safety                           ← Level 2
        └── Hazardous Materials        ← Level 3
              └── OPS_SAF_001          ← Course
```

The taxonomy defines the navigable structure of the repository. It does not change after initial setup without an administrator-controlled migration process.

### 3.2 Unique Course Identifiers

Every course in the repository has a globally unique course identifier. The identifier is the primary key for locating any content: a contributor publishing to `EET1084C` and a user retrieving from `EET1084C` always refer to the same course, regardless of contributor or institution.

Two sourcing rules apply:

1. **Organization-supplied identifiers** — used when the taxonomy provides them natively. In SCNS, `EET1084C` is inherently unique across all institutions and is used directly with no modification.

2. **System-suggested identifiers** — generated when no identifier is supplied by the organization. The system suggests an identifier using the pattern:
   ```
   {LEVEL3_KEY}_{SEQUENTIAL_NUMBER}
   ```
   Example: `HAZMAT_001`, `HAZMAT_002`. The administrator assigns and confirms the identifier; it is immutable once activated.

All course identifiers are:
- Case-insensitive for lookup; stored in uppercase
- Immutable once assigned
- Unique across the entire repository

### 3.3 Taxonomy Configuration

The taxonomy is defined in a JSON configuration file (`taxonomy.json`) supplied at repository setup. The administrator creates this file; a setup wizard may be provided in a future phase to assist with creation.

**Schema:**

```json
{
  "repositoryName": "Florida Engineering Technology Repository",
  "taxonomyVersion": "1.0",
  "identifierSource": "external",
  "levels": [
    { "level": 1, "label": "Discipline" },
    { "level": 2, "label": "Subdiscipline" },
    { "level": 3, "label": "Prefix" }
  ],
  "tree": [
    {
      "key": "ET",
      "name": "Engineering Technologies",
      "children": [
        {
          "key": "EE",
          "name": "Electronic Engineering",
          "children": [
            {
              "key": "EET",
              "name": "Electronic Engineering Technology",
              "courses": [
                {
                  "courseId": "EET1084C",
                  "title": "Survey of Electronics",
                  "creditHours": 3,
                  "curriculumGuideUrl": "https://example.org/curriculum/eet/eet1084c/"
                }
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

**`identifierSource` values:**
- `"external"` — all course identifiers are provided in the taxonomy file (e.g., SCNS)
- `"generated"` — the system assigns identifiers using the suggested pattern; the taxonomy file provides course titles and other metadata but omits `courseId`

The taxonomy file is the authoritative source for course definitions in Phase 1. Courses not present in the file may be added by an administrator in a future phase.

### 3.4 Orphan Containers

At repository startup, the system automatically creates two reserved containers to ensure no content is lost when parent associations are missing or invalid:

1. **Not Classified course** (`_ORPHAN_COURSE`) — houses modules that have no valid associated course in the taxonomy. This container accepts modules published without a valid course identifier or whose parent course is subsequently removed.

2. **Unassigned module** (`_ORPHAN_MODULE`) — within the Not Classified course, houses materials that have no valid associated module. This container accepts materials published without a valid module reference.

These containers are not surfaced in the standard browse interface but are accessible via direct identifier lookup and fully visible in the admin console. They exist to prevent data loss and to provide an administrative resolution queue.

## 4. Content Hierarchy

PreseMaker defines content at three levels. The repository mirrors this structure:

```
Course
  └── Module  (one or more)
        └── Material  (one or more)
```

### 4.1 Course

A Course represents a complete academic or training course associated with a unique course identifier from the taxonomy. It is the top-level organizational container.

| Field | Description |
|---|---|
| Course ID | Globally unique identifier (e.g., `EET1084C`) |
| Title | Short course title |
| Description | Brief course description |
| Taxonomy Path | Level 1 / Level 2 / Level 3 reference |
| Credit Hours | Optional; numeric |
| Curriculum Guide URL | Optional; link to an external course reference |

A course is a container. It does not itself carry instructional files — those live in modules and materials. A course may be published as a complete package (containing modules and materials) or as a metadata-only entry to which modules and materials are later added independently.

### 4.2 Module

A Module is a discrete unit of instruction within a course. It carries its own identity and can be published, retrieved, and reused independently of other modules.

| Field | Description |
|---|---|
| Module ID | System-assigned GUID; unique within the repository |
| Parent Course ID | References the course this module belongs to |
| Title | Module title |
| Description | Brief description |
| Outcomes | List of learning outcomes |
| Topic Hierarchy | Structured list of topics and sub-elements covered |
| Submission Metadata | Contributor, license, submission date, PreseMaker version, status |

### 4.3 Material

A Material is an individual instructional resource within a module. Materials are typed; the declared type governs accepted file formats and any special handling by the repository or client application.

| Field | Description |
|---|---|
| Material ID | System-assigned GUID; unique within the repository |
| Parent Module ID | References the module this material belongs to |
| Material Type | See Section 4.4 |
| Title | Material title |
| Description | Optional |
| File(s) | One or more files; accepted formats are type-dependent |
| Submission Metadata | Contributor, license, submission date, PreseMaker version, status |

### 4.4 Material Types

| Type | Description | Notes |
|---|---|---|
| Presentation | Slide-based presentation | |
| NarratedPresentation | Presentation with embedded audio | |
| InteractiveHTML | Interactive HTML and JavaScript content | May require special rendering handling |
| PrintableHTML | Print-formatted HTML | |
| GutenbergHTML | Chunked Gutenberg-style HTML | |
| Document | Text-based document | Markdown, plain text, PDF |
| Image | Image file or collection | |
| Audio | Audio narration or recordings | |
| Assignment | Assessment or assignment materials | |
| Other | Unclassified materials | Validated against configurable allowlist |

The type field is a string. The repository validates against a configured list but is designed to accept new type values as PreseMaker evolves, without requiring schema changes to existing content.

## 5. Publish and Retrieve Granularity

The repository supports publish and retrieve operations at all three content levels independently.

### 5.1 Publish Levels

| Level | What is Published | Minimum Requirement |
|---|---|---|
| Course | Course metadata; optionally includes one or more modules, each optionally including materials | Valid course ID in taxonomy |
| Module | Module metadata and materials; associated with a course ID | Valid course ID; at least one material |
| Material | Single material file and metadata; associated with a module ID | Valid module ID; at least one file |

A course-level publish is a convenience operation that submits course metadata, one or more modules, and their materials in a single request. Individual module and material publishes allow incremental contribution to an existing course.

### 5.2 Retrieve Levels

| Level | What is Retrieved | Download Format |
|---|---|---|
| Course | Full course package — all published modules and their materials | ZIP |
| Module | Single module — all published materials | ZIP |
| Material | Single material file | Native file format |

Partial retrieval (selecting specific modules within a course, or specific materials within a module) is supported via the browse interface and the PreseMaker client application.

## 6. User Roles

### 6.1 Anonymous Visitor
- Browse all published content at course, module, and material levels
- Search and filter by taxonomy node, course ID, material type, and recency
- Download at any level (course ZIP, module ZIP, individual material)
- Report content as inappropriate (no account required)

### 6.2 Registered Contributor
- All Anonymous Visitor capabilities
- Publish at course, module, or material level
- Update or retract own submissions
- Manage account profile and stored credentials

### 6.3 Administrator
- All Contributor capabilities
- Review flagged content and take action
- Manage contributor accounts (suspend, reinstate)
- View platform statistics
- Access and manage orphan containers

## 7. Browse and Discovery Requirements

### 7.1 Repository Home Page
- Recently added feed (configurable count; default 10 most recent submissions)
- Browse by taxonomy — Level 1 grid navigating to Level 2, then Level 3, then courses
- Global search bar (searches titles, descriptions, course IDs, topic hierarchies)
- Link to contributor registration and login

### 7.2 Taxonomy Navigation Pages
- Level 1 page: shows Level 2 children with submission counts
- Level 2 page: shows Level 3 children with submission counts
- Level 3 page: shows all courses that have at least one published module; submission count and newest date per course

### 7.3 Course Landing Page
- Lists all published modules for the course
- Each module entry: title, contributor, institution (if applicable), submission date, material types available
- Full course package ZIP download
- Link to curriculum guide if defined in taxonomy

### 7.4 Module Detail Page
- Module metadata: title, description, outcomes, topic hierarchy
- List of materials with type badges and file sizes
- Per-material download and full module ZIP download
- Contributor attribution, license badge, submission date

### 7.5 Material Detail Page
- Material metadata and type
- Download button
- Inline preview where the type supports it (HTML types rendered in iframe; images displayed inline)
- Contributor attribution, license badge, submission date

### 7.6 Common Features Across All Detail Pages
- Report button (no account required; single action to flag)
- License badge with link to CC license text
- Curriculum guide link where applicable

## 8. Contributor Account Requirements

- Registration with email confirmation; `.edu` domain detection sets institution display name
- Email confirmation required before first publish
- Password reset via email
- Profile: username, optional display name, email, institution (auto-detected on registration)
- My Submissions: lists own courses, modules, and materials with status; edit, update, and retract actions per item

## 9. PreseMaker Client Features

### 9.1 Repository Settings
- Configurable repository base URL (supports staging vs. production endpoints)
- Stored contributor credentials (encrypted using platform-appropriate secure storage)

### 9.2 Publish Feature
Publish is available at course, module, and material levels:

- **Course publish**: submits a course package from a PreseMaker course definition; may include all modules and materials
- **Module publish**: submits an individual module to an existing course
- **Material publish**: submits an individual material to an existing module

All publish dialogs include:
- Taxonomy course ID entry with real-time validation against the repository
- File/artifact selector for the publish level
- License selector (fixed CC list)
- Contributor authentication status indicator
- Update and retract actions for previously published content
- Progress indicator with cancellation support

### 9.3 Browse and Retrieve Feature
- Taxonomy navigation mirroring the repository browse structure
- Search by course ID or keyword
- Filter by material type; sort by newest
- Download at course, module, or material level into a designated PreseMaker import folder
- Option to create a new PreseMaker course or module pre-populated with downloaded content
- Suggested courses: when no search term is active, surfaces course IDs from the user's existing PreseMaker modules

## 11. Moderation and Trust

- Content publishes immediately upon successful upload; no pre-publication review
- Any visitor may report a module or material (no account required)
- One report flags the content for administrator review
- Flagged content remains publicly visible until an administrator acts
- Administrator actions: Clear Flag (re-publish as clean), Remove (retract), or Contact Contributor
- Contributors whose content is removed receive email notification
- Flag counts are visible to administrators only; the public sees no flag indicator
- Contributor accounts may be suspended by an administrator for repeated violations

## 12. Non-Taxonomy Course Handling — Phase 2

Phase 1 restricts publishing to course IDs present in the configured taxonomy. Phase 2 will introduce:

- System-suggested course ID generation for courses not in the taxonomy, using the `{LEVEL3_KEY}_{SEQUENTIAL_NUMBER}` pattern
- AI-assisted taxonomy mapping suggestion at publish time — given a course title and description, the system suggests the closest existing taxonomy node for the contributor to confirm or override
- A review queue for submissions with unconfirmed taxonomy mappings

---

## 13. Out of Scope — Phase 1

- Artifact versioning beyond contributor-initiated update or overwrite
- Peer review or editorial workflow
- LMS integration (LTI, SCORM)
- Institutional bulk publishing
- RSS or email digest subscriptions
- Comments or discussion on submissions
- Taxonomy restructuring after initial setup (migration path is a Phase 2 concern)
- Multi-repository federation
- Non-CC licensing options
